# ADR-001: File Storage — Cloudflare R2 over Supabase Storage

## Status: Accepted

## Date: 2024-03-10

## Context

Music portal requires storage for audio files (MP3, WAV stems) and cover images.
Initial consideration was Supabase Storage as it integrates natively with the existing
Supabase setup. However, the application expects heavy read traffic — every playback
is a file download — which makes egress costs a critical factor.

Supabase Storage runs on AWS S3 under the hood and bills egress on every stream.
At ~100MB per upload (master + stems + cover) with regular playback traffic:

- Pro plan includes only 100GB storage and 250GB bandwidth
- At 100 active uploaders/day: ~300GB new storage/month
- Egress at moderate playback volume: ~$112/month overage on top of $25 plan

## Decision

Use Cloudflare R2 for all file storage — audio files and cover images.
Supabase remains for database and auth only.

**Architecture split:**

- Supabase — profiles, songs, stems tables + auth
- Cloudflare R2 — bucket `songs` (audio, private), bucket `covers` (images, public)

**R2 pricing at same scale:**

- Storage: ~300GB × $0.015 = ~$4.50/month
- Egress: $0 — R2 has no egress fees, ever
- Operations: negligible at this scale

## Implementation

- R2 is S3-compatible — uses `@aws-sdk/client-s3` with custom endpoint
- Audio files served via presigned URLs (expire 1h) — controlled access
- Cover images served via public R2.dev subdomain — no auth needed
- File paths stored as relative in DB (`user-id/uuid.ext`) — provider-agnostic
- Full URL constructed at runtime via `getCoverUrl()` / `getSongUrl()` helpers

## Alternatives Considered

**Supabase Storage**

- Native integration, no extra config
- Rejected: egress billed per stream, costs scale linearly with playback traffic

**AWS S3**

- Industry standard, mature tooling
- Rejected: same egress cost problem as Supabase Storage, more complex setup

## Consequences

- Extra dependency: `@aws-sdk/client-s3`, `@aws-sdk/s3-request-presigner`
- Two separate services to manage (Supabase + Cloudflare)
- If migrating storage provider in future: change one env variable (`NEXT_PUBLIC_R2_COVERS_URL`, `R2_ACCOUNT_ID`) — no DB updates needed due to relative path pattern
- Upload requires manual rollback on DB error — no shared transaction between R2 and Supabase (see INBOX: R2 + Supabase Upload — Transaction Safety)
